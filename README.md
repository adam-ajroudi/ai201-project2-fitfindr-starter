# FitFindr

FitFindr is a multi-tool AI agent that helps users find secondhand fashion pieces and figure out how to style them with their existing wardrobe. It orchestrates four tools in response to a natural language query — searching listings, suggesting outfits, generating a shareable caption, and assessing whether the price is fair.

---

## Tool Inventory

### `search_listings`
- **Inputs**:
  - `description` (str): Keywords describing the desired item (e.g. "vintage graphic tee").
  - `size` (str or None): Optional size string, case-insensitive (e.g. "M", "W28").
  - `max_price` (float or None): Optional maximum price ceiling.
- **Outputs**: `list[dict]` — matching listings sorted by keyword-overlap relevance score. Each dict contains: `id` (str), `title` (str), `description` (str), `category` (str), `style_tags` (list[str]), `size` (str), `condition` (str), `price` (float), `colors` (list[str]), `brand` (str|None), `platform` (str).
- **Purpose**: Retrieves relevant thrift listings from the dataset based on the user's natural language criteria.

### `suggest_outfit`
- **Inputs**:
  - `new_item` (dict): The selected listing dictionary.
  - `wardrobe` (dict): The user's wardrobe dictionary, containing an `items` key (list[dict]). Each wardrobe item has: `id`, `name`, `category`, `colors`, `style_tags`, `notes`.
- **Outputs**: `str` — a 1–2 paragraph outfit suggestion generated by the LLM. References specific wardrobe pieces by name if wardrobe is non-empty; falls back to general styling advice if wardrobe is empty.
- **Purpose**: Generates complete outfit ideas by combining the new find with the user's existing closet.

### `create_fit_card`
- **Inputs**:
  - `outfit` (str): The outfit suggestion string from `suggest_outfit`.
  - `new_item` (dict): The selected listing dictionary.
- **Outputs**: `str` — a 2–4 sentence social media caption (Instagram/TikTok style) mentioning item name, price, and platform naturally.
- **Purpose**: Generates a shareable, casual caption describing the thrifted fit.

### `compare_price` *(stretch)*
- **Inputs**:
  - `item` (dict): The selected listing. Must contain `price` (float) and `category` (str).
  - `listings` (list[dict]): The full listings dataset from `load_listings()`.
- **Outputs**: `str` — a verdict string with emoji (🟢/🟡/🔴), explanation, median price, price range, and comparable count for items in the same category.
- **Purpose**: Estimates whether the item's price is a great deal, fair, or pricey relative to similar thrifted items in the dataset.

## Planning Loop

The agent follows a sequential, **conditional** loop in `agent.py` (`run_agent()`):

1. **Parse** — regex extracts `max_price` (e.g. `$30` → `30.0`) and `size` (e.g. `size M` → `"M"`). The remainder, after stripping numeric tokens and filler words, becomes `description`.

2. **Search & Retry (stretch)** — calls `search_listings(description, size, max_price)`:
   - **If results found** → continue to step 4.
   - **If results empty** AND size or price constraints were present:
     - Retry 1: drop size constraint → `search_listings(description, None, max_price)`
     - If still empty, Retry 2: drop all constraints → `search_listings(description, None, None)`
     - Set `session["fallback_message"]` to explain what was loosened.

3. **Early exit** — if results are still empty after all retries:
   - Set `session["error"]` = `"No listings found. Try broader keywords or remove size/price filters."`
   - **Return immediately** — `suggest_outfit`, `create_fit_card`, and `compare_price` are **not called**.

4. **Select item** — `session["selected_item"] = results[0]` (top relevance score).

5. **Suggest outfit** — `suggest_outfit(selected_item, wardrobe)` → stored in `session["outfit_suggestion"]`.

6. **Create fit card** — `create_fit_card(outfit_suggestion, selected_item)` → stored in `session["fit_card"]`.

7. **Price check (stretch)** — `compare_price(selected_item, all_listings)` → stored in `session["price_comparison"]`.

8. **Return** the populated `session` dict.

The key adaptive behavior: if `search_listings` returns nothing, the agent does not proceed. It does not call `suggest_outfit` with empty input — it either retries with looser constraints or halts with an actionable message.

## State Management

State is managed by a `session` dictionary instantiated via `_new_session()` at the start of each query:

| Field | Set when | How it flows |
|-------|----------|--------------|
| `session["parsed"]` | After regex parsing | Diagnostic — not passed to tools |
| `session["search_results"]` | After `search_listings` | Stored for reference |
| `session["selected_item"]` | `results[0]` | Passed directly to `suggest_outfit`, `create_fit_card`, `compare_price` |
| `session["outfit_suggestion"]` | After `suggest_outfit` | Passed directly to `create_fit_card` |
| `session["fit_card"]` | After `create_fit_card` | Displayed in Gradio "Fit card" panel |
| `session["price_comparison"]` | After `compare_price` | Displayed in Gradio "Price check" panel |
| `session["fallback_message"]` | When retry fires | Prepended to listing display in UI |
| `session["error"]` | On early exit | Replaces all panel output in UI |

No data is re-entered by the user between steps. The item returned by `search_listings` flows directly into `suggest_outfit` as `session["selected_item"]` — the user never touches it.

## Error Handling

| Tool | Failure mode | What the agent does |
|------|-------------|---------------------|
| `search_listings` | No listings match the query (e.g. "designer ballgown size XXS under $5") | Retries with loosened constraints (drop size, then drop all). If still empty, sets `session["error"]` = "No listings found. Try broader keywords or remove size/price filters." Halts — downstream tools not called. |
| `suggest_outfit` | Wardrobe is empty (`wardrobe['items'] == []`) | Detects empty list; prompts LLM for general styling advice for the item instead of specific wardrobe pairings. Returns a non-empty string — does not crash. |
| `create_fit_card` | Outfit string is empty or whitespace-only | Guard clause at top of function; returns `"Error: No outfit data provided to generate a fit card."` — does not call LLM, does not raise. |
| `compare_price` | Item has no `price` field | Returns `"Unable to assess price — no price data available for this listing."` |
| `compare_price` | Fewer than 2 comparables in category | Returns `"Not enough comparable listings in the '<category>' category to assess pricing."` |

**Concrete example tested:** Running `search_listings("designer ballgown", size="XXS", max_price=5)` returns `[]`. Running the full agent with this query sets `session["error"]` and returns — `session["fit_card"]` remains `None`, `session["outfit_suggestion"]` remains `None`. The UI displays the error message across all three output panels without crashing.

## Stretch Features

### Price Comparison Tool (+2pts)
`compare_price(item, listings)` compares the item's price against all other listings in the same category. It computes the median price of comparables, classifies the item as a great deal (>20% below median), fair price (within ±20%), or pricey (>20% above). Result is shown in a dedicated "Price check" panel in the UI.

**How comparisons are made:** The tool filters `load_listings()` to items with the same `category` value, excludes the item itself (matched by `id`), and requires at least 2 comparables. Statistics are computed directly in Python — no LLM call, purely data-driven.

### Style Profile Memory (+2pts)
Users can add wardrobe items via the "My Style Profile" tab in the UI. Items are saved to `data/style_profile.json` and persist across sessions. Selecting "Use saved profile" on the Search tab loads the saved wardrobe as the agent's wardrobe input. Past searches are also stored (capped at 20) and displayed in the profile tab.

**Storage approach:** `load_style_profile()` and `save_style_profile()` in `utils/data_loader.py` read/write `data/style_profile.json`. If the file doesn't exist, a default empty profile is returned. The profile schema: `{"preferred_styles": [], "preferred_sizes": [], "past_searches": [], "saved_wardrobe": {"items": []}}`.

### Retry Logic with Fallback (+1pt)
If `search_listings` returns no results and size or price constraints were specified, the agent automatically retries by first dropping the size filter, then dropping all constraints. The UI surfaces a `⚠️` warning explaining what was adjusted (e.g. "No listings found for size M. Showing results in other sizes.").

## Spec Reflection

**One way planning.md helped during implementation:**
Writing out the state management table explicitly — which field is set when, and what it flows into — made it immediately clear that `session["selected_item"]` had to be set before calling both `suggest_outfit` and `create_fit_card`. This prevented a design where I might have passed `results[0]` directly without storing it, which would have broken state traceability.

**One divergence from the spec, and why:**
The initial spec assumed only three tools. During implementation it became clear that price assessment added real value and could be done without an LLM call (purely data-driven with the existing dataset), making it a cheap addition. The architecture diagram was updated before implementation to include `compare_price` as a final step that never causes early exit — it always runs if a listing was found, so it couldn't block the main flow.
